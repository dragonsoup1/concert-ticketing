# 요구사항 분석

## 프로젝트 개요

대규모 트래픽 환경에서 콘서트 티켓 예매를 처리하는 백엔드 시스템. 동시 접속자 급증 시에도 좌석 중복 예약 없이 데이터 일관성을 보장하며, PG사 장애에도 서비스가 중단되지 않아야 한다.

---

## 기능 요구사항

### 콘서트 조회

| ID | 요구사항 |
|----|---------|
| F-01 | 예정된 콘서트 목록을 페이지네이션으로 조회할 수 있다 (기본 20건) |
| F-02 | 특정 콘서트의 상세 정보(제목, 장소, 일시, 총 좌석 수)를 조회할 수 있다 |
| F-03 | 특정 콘서트의 잔여 좌석 목록(번호, 등급, 가격, 상태)을 조회할 수 있다 |

### 가상 대기열

| ID | 요구사항 |
|----|---------|
| F-04 | 사용자는 콘서트별 대기열에 입장 토큰을 발급받을 수 있다 (중복 발급 불가) |
| F-05 | 사용자는 현재 대기 순위와 예상 대기 시간을 조회할 수 있다 |
| F-06 | 시스템은 1초마다 최대 50명씩 대기열에서 배치 입장 허가를 부여한다 |
| F-07 | 입장 허가는 10분간 유효하며, 만료 후에는 대기열에 다시 진입해야 한다 |

### 좌석 예약

| ID | 요구사항 |
|----|---------|
| F-08 | 입장 허가를 받은 사용자만 좌석을 예약할 수 있다 |
| F-09 | 예약 요청 시 좌석은 즉시 임시 선점(HELD) 처리된다 |
| F-10 | 예약은 생성 후 5분간 유효하며, 결제 미완료 시 자동 만료(EXPIRED)된다 |
| F-11 | 만료된 예약의 좌석은 자동으로 AVAILABLE 상태로 반환된다 (30초 주기 스케줄러) |
| F-12 | 사용자는 PENDING 또는 CONFIRMED 상태의 예약을 취소할 수 있다 |
| F-13 | 예약 취소 시 좌석은 AVAILABLE 상태로 반환된다 |

### 결제

| ID | 요구사항 |
|----|---------|
| F-14 | 사용자는 예약에 대한 결제를 요청할 수 있다 |
| F-15 | 결제 성공 시 예약 상태가 CONFIRMED → PAID로 전환되고 좌석이 SOLD 처리된다 |
| F-16 | PG사 장애 시 결제 상태가 PENDING_CONFIRMATION으로 기록되어 서비스가 중단되지 않는다 |
| F-17 | 동일한 Idempotency-Key로 중복 결제 요청 시 최초 결제 결과를 그대로 반환한다 |

### 이벤트 발행

| ID | 요구사항 |
|----|---------|
| F-18 | 예약 생성, 결제 완료, 예약 만료 시 Kafka로 도메인 이벤트를 발행한다 |
| F-19 | Kafka 발행 실패 시 최대 5회 재시도하며, 이후에는 FAILED로 마킹한다 |

## 비기능 요구사항

### 동시성 제어

| ID | 요구사항 | 구현 |
|----|---------|------|
| NF-01 | 동일 좌석에 대한 동시 예약 요청이 발생해도 1건만 성공해야 한다 | Redisson 분산 락(leaseTime 3s) + JPA `@Version` Optimistic Lock |
| NF-02 | JVM 노드가 여러 개인 환경에서도 동시성이 보장되어야 한다 | Redisson RLock (Redis 기반 분산 락) |

### 처리량 및 성능

| ID | 요구사항 | 구현 |
|----|---------|------|
| NF-03 | 사용자당 초당 최대 10 req를 초과하는 요청은 HTTP 429로 차단한다 | Redis ZSet 슬라이딩 윈도우 Lua 스크립트 |
| NF-04 | 콘서트/좌석 조회는 캐시를 통해 DB 부하를 줄인다 | Redis @Cacheable (콘서트 5분, 좌석 60초) |

### 가용성

| ID | 요구사항 | 구현 |
|----|---------|------|
| NF-05 | PG사 장애 시 Circuit Breaker가 작동해 서비스가 중단되지 않아야 한다 | Resilience4j (실패율 50%, 대기 10s, 반개방 3회) |
| NF-06 | PG 응답 지연이 3초를 초과하면 타임아웃으로 처리한다 | Resilience4j TimeLimiter |

### 데이터 일관성

| ID | 요구사항 | 구현 |
|----|---------|------|
| NF-07 | Kafka 이벤트 발행은 at-least-once를 보장해야 한다 | Transactional Outbox 패턴 |
| NF-08 | 이벤트 발행과 도메인 저장은 항상 동일 트랜잭션 내에서 처리되어야 한다 | `Propagation.MANDATORY` |

### 멱등성

| ID | 요구사항 | 구현 |
|----|---------|------|
| NF-09 | 동일한 결제 요청이 여러 번 들어와도 단 1회만 처리되어야 한다 | Redis 캐시(24h) + DB unique constraint 이중 가드 |
| NF-10 | Redis 재시작 후에도 멱등성이 보장되어야 한다 | DB `payment.idempotency_key` unique constraint (cold-start fallback) |

### 접근 제어

| ID | 요구사항 | 구현 |
|----|---------|------|
| NF-11 | 대기열 입장 허가 없이 예약 요청을 보내면 거부되어야 한다 | `QueueTokenValidationFilter` |

## 예약 상태 머신

```
PENDING ──(5분 TTL, 스케줄러)──► EXPIRED  (좌석 AVAILABLE 반환)
   │
   ├──(POST /payments 성공)──► CONFIRMED ──(PG webhook)──► PAID
   └──(DELETE /reservations)──► CANCELLED  (좌석 AVAILABLE 반환)
```

## 결제 상태

| 상태 | 설명 |
|------|------|
| REQUESTED | 결제 요청 생성됨 |
| SUCCESS | PG 승인 완료 |
| FAILED | PG 거절 또는 오류 |
| PENDING_CONFIRMATION | Circuit Breaker 작동, PG 응답 대기 중 |

## 좌석 상태

| 상태 | 설명 |
|------|------|
| AVAILABLE | 예약 가능 |
| HELD | 임시 선점 (예약 PENDING 중) |
| SOLD | 결제 완료 |
